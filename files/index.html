Легко воспроизводимая инфраструктура сервера – подход IaC

Установка необходимых пакетов описана в playbook ansible, что позволяет хранить конфигурацию host'а в репозитории в виде кода. Так же это дает возможность быстро развернуть идентичную инфраструктуру на другом железе, в случае необходимости. Что касается этого проекта, то для него требуется установка docker engine и выдача прав пользователям на исполнение dockercli.

Конфигурация домашней LAN

Без серьезных финансовых затрат на оборудование нельзя применить множество практик по резервированию, масштабированию и повышению безопасности сети, но для небольшого домашнего сервера с ограниченным кругом пользователей этого и не требуется. А для всего остального было выполнено следующее:
От провайдера по PPPoE динамически выдается публичный IP-адрес. Так как адрес динамический, а статический можно получить за отдельную сумму, пришлось прибегнуть к использованию dynamicDNS CloudFlare. Это позволяетавтоматически обновлять конфигурацию DNS, конкретнее соответствующий домену IP-адрес, в случае выдачи провайдером нового адреса.
Для доступа к серверу из «всемирной сети» на домашнем роутере были открыты 80 и 443 порты, соответствующие протоколам HTTP и HTTPS. Внутри домашней сети используется приватная сеть за NAT. Чтобы перенаправить приходящие запросы по HTTP/HTTPS на сервер были проброшены указанные ранее порты (PAT) на приватный IP-адрес сервера.
Доменное имя зарегистрировано бесплатно через freenom.com в доменной зоне .ml

Pipeline CI/CD на Jenkins

Для организации непрерывных интеграции и доставки кода используется Jenkins.
При выполнении «push» в master ветку github оповещает сервер Jenkins. Тот в свою очередь инициирует выполнение job, который скачавает проект на сервер, выполняет на нем сборку нового docker образа, а затем обновляет выполняемый контейнер до актуальной версии.
Но здесь есть нюанс. Jenkins развернут в контейнере. Поэтому для сборки образов и работы с docker используется подключение по ssh к хосту. Это необходимо, чтобы обойти проблему dind.
